Oui, là je vois le vrai problème : **ce n’est pas ta VM**. C’est encore le script.

Dans `Install-AdcsCertificationAuthority`, ton script passe :

```powershell
ValidityPeriod = 'Years'
ValidityPeriodUnits = 5
```

Or pour une **CA subordonnée**, Microsoft dit explicitement de **ne pas utiliser ces paramètres**, car la durée du certificat de CA est déterminée par la CA parente qui signe la CSR. C’est exactement ce qui provoque ton erreur `ERROR_CLUSTER_PARAMETER_MISMATCH`. ([Microsoft Learn][1])

J’ai corrigé ça dans une v3 :
[run-on-windows-fixed-apv-v3.ps1](sandbox:/mnt/data/run-on-windows-fixed-apv-v3.ps1)

Fais ça dans ta VM :

```powershell
cd C:\Users\Administrateur\Desktop\6.3-Gestion-Certifs\windows-adcs

Copy-Item .\run-on-windows-fixed-apv-v3.ps1 .\run-on-windows.ps1 -Force

Remove-Item C:\TP-Crypto-ADCS -Recurse -Force -ErrorAction SilentlyContinue

Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass -Force

.\run-on-windows.ps1
```

Cette fois, l’étape attendue est qu’il crée :

```text
C:\TP-Crypto-ADCS\output\sub-adcs-ca.req
```

Si ça arrive jusque-là puis qu’il dit que le certificat signé est absent, **c’est normal** : il faudra signer la CSR côté Linux/OpenSSL, puis mettre le certificat signé dans :

```text
C:\TP-Crypto-ADCS\input\sub-adcs-ca-signed.crt
```

et le certificat racine dans :

```text
C:\TP-Crypto-ADCS\input\root-ca.crt
```
